# 高性能路网加载与最短路计算核心算法说明

本文面向不熟悉路网计算的同事，解释当前项目中两个核心模块的设计思路。

## 1. 路网加载模块

核心类：

```python
RoadNetworkLoader
```

### 1.1 输入是什么

输入是 CSV 文件，列名可以是 `WKT` 或 `wkt`。每一行类似：

```text
LINESTRING(100.1 13.7,100.2 13.8)
```

这里的坐标顺序是：

```text
经度 纬度
```

一条 `LINESTRING` 表示一段道路几何。如果里面有多个点：

```text
LINESTRING(A,B,C,D)
```

系统会把它拆成：

```text
A-B
B-C
C-D
```

这些相邻点之间的线段就是路网图里的边。

### 1.2 路网为什么要变成“图”

最短路算法处理的不是 WKT 字符串，而是图：

- 路口或道路形状点：节点。
- 两个相邻点之间的道路：边。
- 边的长度：权重。

可以理解为：

```text
城市地图 -> 节点和边组成的数学图
```

### 1.3 节点去重

CSV 中很多道路会共享同一个坐标点。系统会把相同坐标合并成同一个节点。

实现方式是把经纬度放大后转成整数 key：

```python
key = (round(lon * 1e7), round(lat * 1e7))
```

这样可以避免浮点数直接比较带来的误差，也能快速查重。

### 1.4 边长计算

经纬度不是平面坐标，所以系统使用 Haversine 公式计算两点之间的球面距离，单位是米。

这比简单的欧氏距离更适合地理坐标。

### 1.5 CSR 压缩图结构

加载完成后，系统不会用大量 Python 对象保存邻接表，而是转成数组：

```text
offsets
neighbors
weights
```

这叫 CSR 结构，可以理解为“压缩版邻接表”。

优点：

- 内存占用更低。
- 连续数组遍历更快。
- 适合百万级边数据。

### 1.6 最近节点索引

用户输入的起点和终点通常不会刚好落在路网节点上，所以系统需要找到最近的路网节点。

实现使用：

```python
scipy.spatial.cKDTree
```

它可以很快完成最近邻查询，通常是毫秒级。

### 1.7 加载模块优点

- 支持百万级 WKT CSV。
- 流式读取 CSV，不一次性把所有文本读进内存。
- 节点自动去重。
- 使用数组结构降低内存占用。
- 自动建立最近节点索引。
- 额外保存无向边数组，支持前端按地图视野局部加载路网。

## 2. 最短路计算模块

核心类：

```python
RoadDistanceCalculator
```

### 2.1 查询流程

给定：

```text
起点经纬度
终点经纬度
```

系统执行：

1. 找到离起点最近的路网节点。
2. 找到离终点最近的路网节点。
3. 如果任一点离路网超过 1km，直接失败。
4. 在路网图中计算两个吸附节点之间的最短路径。
5. 返回距离、路径坐标和耗时。

### 2.2 为什么要吸附

用户输入的是任意经纬度，而路网算法只能在图节点之间寻路。

所以需要：

```text
真实输入点 -> 最近路网节点
```

这个过程叫吸附。

### 2.3 为什么设置 1km 阈值

如果输入点离路网太远，强行吸附会得到错误路线。

例如用户上传的是曼谷路网，却输入了清迈坐标。如果不限制，系统会吸附到曼谷某个最近点，然后给出没有意义的路径。

因此当前规则是：

```text
起点或终点离最近路网节点超过 1000m，则不寻路
```

### 2.4 双向 Dijkstra

当前最短路算法是双向 Dijkstra。

普通 Dijkstra 是：

```text
从起点一圈圈往外扩展，直到找到终点
```

双向 Dijkstra 是：

```text
起点往外扩展
终点也往外扩展
两边搜索相遇后得到最短路
```

相比单向 Dijkstra，双向搜索通常能减少扩展范围。

### 2.5 路径还原

搜索时系统会记录每个节点是从哪个前驱节点来的。

找到最短路后，通过前驱数组把完整路径还原成：

```text
[(lon, lat), (lon, lat), ...]
```

前端地图就是用这个坐标序列画出蓝色路径。

### 2.6 当前算法优点

- 实现稳定，适合正权无向路网。
- 不需要复杂预处理。
- 中短距离查询性能较好。
- 与 CSR 图结构配合，内存和遍历效率都比普通 Python 邻接表好。
- 支持返回完整路径，方便前端可视化。

### 2.7 当前算法限制

双向 Dijkstra 对长距离查询仍然会扩展大量节点。

在 300 万节点、300 万边级别数据上，本轮测试结果：

```text
约 10km：约 0.27s 算法耗时
约 20km：约 0.80s 算法耗时
约 30km：约 1.70s 算法耗时
约 50km：约 4.34s 算法耗时
```

所以当前方案已经适合百万级加载和中短距离查询，但长距离查询还需要进一步优化。

## 3. 后续性能优化方向

### 3.1 A*

A* 会利用“当前位置到终点的直线距离”作为启发式，优先搜索更可能接近终点的方向。

优点：

- 实现难度中等。
- 对中长距离通常比 Dijkstra 更快。

### 3.2 ALT Landmarks

ALT 会预先选一些地标节点，计算地标到所有节点的距离。查询时利用三角不等式估计方向，减少无效搜索。

优点：

- 查询速度更好。
- 适合大规模静态路网。

代价：

- 需要预处理。
- 需要额外存储 landmark 距离数组。

### 3.3 Contraction Hierarchies

这是工业级路网寻路常用方案之一。

优点：

- 查询极快。

代价：

- 预处理复杂。
- 实现成本明显高于 Dijkstra 和 A*。

## 4. 一句话总结

当前系统的核心思路是：

```text
把 WKT 路网 CSV 快速转换成压缩图结构，
用 KDTree 快速把用户坐标吸附到路网节点，
再用双向 Dijkstra 在图上计算最短路。
```

它的优势是结构清晰、加载快、内存相对可控、中短距离查询稳定；主要短板是超大路网下的长距离查询还需要 A*、ALT 或 CH 这类预处理算法继续优化。
