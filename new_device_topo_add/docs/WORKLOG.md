# 工作迭代与思考记录（复盘用）

> 项目：运营商接入层新增 CSG 组网优化
> 目的：记录需求澄清、建模决策、算法实现的每一次迭代与背后的思考，便于后续复盘总结。

---

## 0. 任务背景

运营商接入层网络新增了大量 **孤点 CSG**（无 IP 连接）。需为其添加 IP 连接，接入现网并具备可靠性，
结合 **GIS 经纬度** 与 **路网距离（road_distance.csv）** 尽量就近连接以降本。
目标组网：① 每个设备有两个方向能到达 ASG；② 基于 ASG/CSG 形成的环尽量 ≤10 跳。

---

## 1. 需求澄清（不猜意图，先对齐口径）

### 第一轮：场景理解 + 4 个结构化决策

发现工作目录为空（无真实数据），桌面其它目录的 device.csv 角色是 PE/Core，与本任务（csg/asg）不符，
因此**没有直接拿现成文件当任务数据**。先和用户确认 4 个会实质改变建模的问题，得到口径：

| 问题 | 用户口径 | 对建模的影响 |
|---|---|---|
| "≤10 跳"指什么 | **整个环节点数 ≤ 10** | 环长 = CSG 数 + 锚 ASG 数 ≤ 10，转化为容量约束 |
| "两个方向到 ASG" | **两条边不相交路径即可**（可同 ASG） | 等价"到 ASG 集合 2-边连通"，用超汇 max-flow≥2 判定；允许单 ASG 闭合环 |
| 改动范围 | **只加链路，可插入现有环** | 现网不重规划，孤点可 splice 进现有环 |
| 本轮交付 | **先出算法设计方案** | 第一阶段产物为设计文档 |

### 第二轮：数据与成本细节

| 问题 | 用户口径 |
|---|---|
| 真实数据 | 无，**需自行构造数据穿刺**；road_distance 字段 = Src/Sink/Distance |
| road 覆盖 | **仅 CSG-CSG 且直线 ≤3km**；其余用直线距离 |
| 距离阈值 D_max | 不写死，结合分布与拓扑自适应 |
| 新增 CSG 链路数 | **2 条**，可连 ASG 也可连 CSG（度=2） |
| splice 成本 | **只算新增光缆**，断开旧光缆不计 |

**思考**：口径②选"边不相交路径"而非"必须双归不同 ASG"，使**单 ASG 闭合环**成为合法解，
显著扩大可行域、降低成本。这是后续建模的关键简化。

## 2. 建模（算法设计方案.md）

核心洞察：**口径② = 把节点放到"经过 ASG 的环"上**（环上每点沿两方向各有一条到 ASG 的边不相交路径）；
口径① = 环容量上限。问题归约为 **容量受限的成环网络设计（m-Ring-Star / SONET 环装载变体，NP-hard）**。

- 成本：路网优先，缺失用直线（Haversine）兜底。
- 目标：min 新增光缆总长。
- 方案主线：候选边稀疏化(kNN) → 构造启发式(最便宜插入/聚类成环) → 局部搜索 → 超汇 max-flow 可行性校验 →（可选）MILP。

## 3. 数据构造（gen_data.py）

合成一套贴近 IP RAN 的拓扑：5 ASG + 8 现有环（部分留余量、部分接近满）+ 40 孤点
（20 个贴近现网 → 应被插环；20 个远端成 4 簇 → 应新建环）。road_distance 仅 CSG-CSG 直线≤3km，
距离×绕行系数(1.2~1.6) 模拟路网。**固定随机种子**保证可复现。
产出：99 设备、62 链路、396 条 road 记录。

## 4. 算法实现的迭代（optimize.py）

### 迭代 ① 首版 → 暴露两个 bug
- **现有环只识别出 4 个**：原因——我把"双 ASG 环"生成成了 `ASG_A—…—ASG_B` 的**开放链**
  （接入层非闭合 cycle，靠核心层闭合），用 `biconnected_components` 只能识别闭合环。
  → **修正**：统一环定义为"去掉 ASG 后的 CSG 连通分量 + 其挂接的 ASG"，
  同时覆盖"单 ASG 闭合环"和"双 ASG 开放链"，二者均满足口径②。这其实更贴近真实 IP RAN 形态。
- **splice 备注解析崩溃**：节点名含 `-`，按 `-` split 出错。→ 改为**结构化记录被替换的边**。

### 迭代 ② 插入现有环安置=0 → 重构优化逻辑
- 现象：40 个孤点全去新建环，"插环"能力没被行使，成本偏高(158km)。
- 根因：用了"插入代价 ≤ 1.6×最近距离"的**魔法闸门**；而插入天然需 2 条光缆 ≈ 2×最近距离，被一律挡掉。
  闸门是"2 条边 vs 1 条边"的不公平比较。
- **修正思路**：不调魔法阈值，而是把"插环 vs 新建环"交给**局部搜索按真实成本决策**。
  把解结构重构为"带锚 ASG 的有序环"，构造后用 **Or-opt 迁移（移到任意更省的环，含现有环空位）+ 环内 2-opt**。
- 结果：成本 158→**146km**，6 个插入现有环 / 34 个进新环，三项硬约束全过。
  6 个插入是经济上理性的结果（散落近端点被拉进相邻现有环）。

**思考**：两次迭代的共性教训——
①"真实形态优先于代码便利"（环定义要贴 IP RAN 而非贴 networkx API）；
② 启发式里**少用魔法阈值，多让局部搜索按目标函数自洽决策**。

### 迭代 ③ 目录归类 + 确定性修复
- 按 `docs/ code/ data/` 归类后重跑，发现**同一固定种子数据、结果却在 146/135km 间漂移**。
- 根因：优化器多处依赖 **Python set 的迭代顺序**（哈希随机化 → 每进程不同），破坏可复现性。
- **修正**：在分量遍历、ASG 选择、贪心聚类等所有 `set` 迭代/`min` 处统一**排序 + (距离,名称) 平局裁决**。
- 结果：连续多次稳定输出 **141.59km**，与 git/复盘目标一致。

**思考**：可复现性是"可回溯"的前提。启发式里凡是遍历 set / 平局，必须显式定序，否则版本对照失去意义。

### 迭代 ⑤ 差异化测试用例 + 不规整拓扑加固
- 需求：构造差异化用例验证算法鲁棒性——①现有环不规整（环上挂链分支/spur）；②新增孤点全部贴近现网。
- **生成器场景化**：`gen_data.py` 增加场景参数（baseline/branched/all_near/...），固定用例映射
  `test_1=baseline, test_2=branched, test_3=all_near`。
- **暴露并修复的健壮性问题**：原 `analyze` 假设每个 CSG 连通分量是简单链（度≤2）做路径回溯；
  分支节点度≥3 会让回溯漏掉分支节点 → 这些边在重建时被丢弃，导致最终图与现网不符。
  - **修正**：分量若含分叉（度≥3）或非双归 → 整分量"**冻结**"：边原样保留、不作插入目标、节点不可迁移；
    `build_final_graph` 显式并入 `frozen_edges`，`orig_edges` 改用**全部现网链路**为权威来源。
- **校验分层**：把"算法责任的硬失败"（新增孤点未双归 / 度≠2 / 算法环>10）与
  "现网既有状况"（如单归 spur）**分开报告**——后者仅提示，不计入算法失败。
- **发现的范围问题（待你确认）**：branched 场景里现网存在**既有单归分支节点**（CSG-B*）。
  当前算法范围只为"孤点(度0)"加链，不主动整改这些既有单归节点。是否需要扩展算法一并为其补第二上联？

**思考**：穿刺的价值正在于"打到假设的边界"。分支场景一次性暴露了"简单链假设"，并把一个**业务范围问题**
（是否整改既有单归）显式化——这类问题宁可报告+确认，不可静默处理。

**已知局限**：当前对含分支的环采取"整分量冻结"，较保守——会让该环的规整环段也暂不可插入。
后续可细化为"抽取 2-边连通环段为可插入 Ring、仅冻结 spur"。

### 迭代 ⑥ 三种组网方式 + 成本择优 + 输入封装 + 全量标注输出
- 需求：①新增组网三方式（fiber_extension 双锚绕接 / loop_insertion 插环 / new_loop 新环到ASG），均双归、按成本自动择优；
  ②输出全量链路并标注原有/新增+方式；③封装输入加载。环长≤10 不再硬约束。
- **统一框架**：聚簇(并查集, 阈值 `D_CLUSTER`) → 簇内成链(最近邻+2-opt) → 对每簇评估三方式"新增光缆成本"取最小可行 → 超汇 max-flow 校验全员双归。
  - fiber_extension：链两端各锚一台在环设备；约束 `MAX_FIBER_CHAIN=3`(链长)、`MAX_NEW_PER_DEVICE=3`(挂接点新增邻居)，均可配置。
  - loop_insertion：splice 进某条环边(断旧边不计成本)。
  - new_loop：两端连最近一对 ASG。
- **输入封装**：`NetworkData` + `load_input(input_dir)` 从指定 input 读 device/link/road_distance 入内存。
- **输出**：`new_links.csv` 改为全量最终链路，列 `Link Type(原有/新增/原有-已splice断开) · Method · Cost · Source`。
- **观察(待确认)**：纯成本下 **new_loop 从未被选中**——现网在环 CSG 比 ASG 密集得多，连到邻近在环设备恒比连到 ASG 对更便宜；
  且环长不约束后出现 18~21 节点的大环。若要让远端簇优先 new_loop，需引入"距现网阈值"规则而非纯成本。

**思考**：把三方式归一为"链的两端接到两个锚点"的统一操作，差别只在锚点来源(在环设备/环边/ASG对)，
使"按成本择优"成为一行比较；但也因此暴露——纯成本目标会架空"为远端设计的 new_loop"，这类策略意图需用约束/规则显式表达。

### 迭代 ⑦ 规则B(远端→new_loop) + 规则C(环长软约束) + 现网单归整改
- 在 v2 基础上落实用户确认的 B+C 组合与"既有单归一并整改"：
  - **规则B**：簇到最近在环设备距离 > `D_FARNESS(3km)` 时强制 new_loop —— 让远端簇成自有环而非长链挂现网。
  - **规则C**：候选成本 += `LAMBDA_RING · max(0, 成环后节点数 − RING_SOFT_MAX)`；用并查集 `RingTracker`
    动态跟踪成环规模(含 loop/fiber 的增长与跨环合并)，抑制大环。
  - **现网整改**：识别既有单归 spur，给自由端补一条到最近"既有双归设备/ASG"的链路闭合成环(方式 `remediation`)。
- **修复的真实 bug**：整改回滚逻辑里，当最近锚点恰是 spur 的现有邻居时，`add_edge` 为 no-op 而 `remove_edge`
  会**误删原边**、污染图，导致该 spur 永远修不好(B006)。修正：跳过已相邻的候选锚点。
- **效果**：new_loop 不再恒为 0（test_1=5、test_2=10）；最大环 20→13(test_1)；test_2 的 6 个既有单归 → 4 条整改链路 → **全员双归**。

**思考**：规则B/C 把"运维合理性"用约束/惩罚显式编码进成本目标，弥补纯成本的盲区；
整改 bug 再次印证"无向图上 add/remove 回滚必须先判存在性"，否则静默吞掉原始拓扑。

### 迭代 ⑧ 大规模穿刺(test_4) + 参数化生成器 + 性能修复
- **生成器参数化**：`gen_data.py` 重写为 `PROFILES`(baseline/branched/all_near/large) + 命令行 `key=value` 覆盖；
  可指定 `n_asg, target_existing, ring_size_*, dual_asg_frac, branch_frac, branch_len_*, n_new, near_frac, region_km, far_clusters` 等。
  road_distance 改用**空间网格**(3km 桶)找邻近对, 避免 O(n²)；4500 节点造数 0.7s, 5.7万 road 记录。
- **test_4(large)**: 120 ASG + 3000 现网CSG + 207 分支CSG + 1200 新增(70%近/30%远), 273 环含 109 环带链。
- **关键性能修复**：原双归校验对每个CSG跑 `networkx.maximum_flow_value` —— 4400 次重型 maxflow, 在大规模下**卡死**。
  改为 **O(V+E) 的2-边连通判定**：把 {超汇t}∪ASG 串成环成2-边连通核, 去桥(`nx.bridges`)后不在 t 分量的 CSG 即单归。
  整改的单点判定同样改用此法。→ test_4 端到端 **33.9s**, 小用例结果与 maxflow 版**完全一致**。
- **结果(test_4, all_pass=True)**: 新增 1839 链路 / 4342.6km; fiber=150/loop=25/new_loop=355/整改=109;
  207 既有单归 → 109 整改链路 → 全员双归。
- **观察**: 最大环 45(250 环>10)。大簇(贴近区 24 节点簇)无法拆分→大环, 软惩罚压不动。需"按容量拆簇"。

**思考**：规模化把"算法正确但用了重型原语"的隐患放大成致命瓶颈；选对等价的轻量判定(2-边连通 vs 最大流)是可伸缩性的关键。

### 迭代 ⑨ 纠偏: 短纤就近入网 + 修复 splice 显示
- 用户反馈：① new_loop 滥用、给贴近现网的新增点拉长纤直连 ASG，不合理；② 可视化里 loop_insertion 的 splice 断边没显示。
- **问题2(显示)定位**：算法层 `broken` 正确；但 `build_viz` 画现网环用的是**未随 splice 更新的环对象**，
  把被断的旧边仍画成实线盖住虚线 → **可视化 bug**。修正：`build_viz` 跳过 `broken` 中的边(只以虚线表示)。
- **问题1(选路)三处根因 + 修复**：
  - 锚点只认规整环 CSG → 环带链被整体冻结、其 1400+ CSG 不可锚 → 贴近分支环的新增点被判远端走 new_loop。
    **修正**：可锚集合改为"全部现网既有双归 CSG"(用 2-边连通求得, 含冻结环的环上节点)；farness 也据此判定。
  - 环长惩罚对本就>10 的现网大环一刀切重罚 → 倒逼 new_loop。**修正**：改**边际惩罚**(只罚推过 max(软上限,当前规模) 的部分) + λ 1500→800。
  - 大的贴近簇(>3)无法 fiber。**修正**：贴近簇拆成 ≤3 子链就近 fiber/插环；远端簇拆 ≤8 成新环。
  - 最近锚/farness 查找用**空间网格**, 控制大规模耗时。
- **效果(test_4)**：可锚 CSG 1773→**3000**；fiber 150→**537** / new_loop 355→**74**；总成本 4342→**2130km(腰斩)**；27s, all_pass。
  小用例 test_3(全贴近)→ 21 条 fiber、零 new_loop。
- **新副作用**：贴近优先使大量 fiber 绕接同一环、撑大 2-边连通域(test_4 最大环 68)。软惩罚压不动, 待"按环容量分流"。

**思考**：选路的"合理性"高度依赖锚点可见性与惩罚的边际化——把冻结环节点纳入锚点、把惩罚改成边际，才让"就近入网"自然胜出。
而 splice 显示 bug 再次说明：可视化必须由"算法的真实最终态"驱动，任何用旧对象近似都会失真。
